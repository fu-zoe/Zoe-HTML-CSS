Chapter 7 Semantics

Semantic Tags & Landmarks

Semantic tags don’t change look by default, but add meaning for browsers, search engines, and assistive tech. Use the toggle to visualize structure.

General idea

  • Semantics = meaning; default visuals are minimal.
  • Helps navigation for screen readers via landmarks (header, nav, main, footer...).
  • Improves SEO and document structure (headings hierarchy).

Live demo (code ↔ render)

<header>
  <h1>Main heading</h1>
  <nav aria-label="site"></nav>
</header>

<main>
  <article id="a1">
    <h2>Article1 heading</h2>

    <section id="s1">
      <h3>Section1 heading</h3>

      <aside>
        <h4>Aside heading</h4>
        <details>
          <summary>Show more details about Section 1</summary>
          inline element mark: <mark>highlighted content</mark>
        </details>
      </aside>
    </section>
  </article>

  <section id="s2">
    <h2 id="section2-heading">section2 heading</h2>
    <nav aria-labelledby="section2-heading"></nav>
  </section>
</main>

<footer>
  <nav aria-label="footer"></nav>
</footer>

Tips & rules

  • Headings: One page-level <h1>; then descend in order (h2 inside article/section, etc.).
  • section vs article: Article is self-contained (blog post, card, comment). Section groups related content inside a page region. Prefer a heading for each section.
  • Multiple navs: Differentiate with aria-label or aria-labelledby (e.g. site nav vs in-section nav).
  • div / span: No semantics. Use for styling hooks or when no semantic element fits.
  • details/summary: Built-in disclosure widget; keep summary clear and focusable.
